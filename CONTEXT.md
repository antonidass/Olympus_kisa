## Правила написания сценариев

- Длина: история за ~1 минуту (около 150–200 слов)
- Язык: живой, разговорный русский — без канцелярита и ИИ-шаблонов
- Структура: крючок в первой фразе → развитие событий → неожиданный или эмоциональный финал
- Тон: лёгкий, с иронией, но уважительный к мифу
- Имена: греческое написание (Прометей, Эпиметей, Гефест, Зевс и т.д.)

### Кликбейтный хук после интро (ОБЯЗАТЕЛЕН)

Каждый `prompts/voiceover.md` строится по жёсткому шаблону:

```
# <Название мифа>

<Имя/Название>. Миф за минуту.

<КЛИКБЕЙТНЫЙ ХУК — 1–2 предложения, удерживают зрителя в первые 3 секунды>

<основной текст истории, ~150–200 слов>
```

Хук идёт **сразу после** строки «<...>. Миф за минуту.» и **до** основного повествования. Без него ретеншн рассыпается на 2-й секунде — зритель не понимает, зачем смотреть, и свайпает. Эталон — Мидас:

```
Царь Мида́с. Миф за минуту.

Представьте: всё, чего касаетесь, становится золотом. Царь Мида́с думал — это мечта. Оказалось — ловушка.
```

Что даёт «Представьте: всё, чего касаетесь, становится золотом» в первые 3 секунды:
- ставит зрителя внутрь ситуации (приём «представьте»)
- описывает ставку максимально кликбейтно (все мечтают о золоте → крючок узнавания)
- сразу обещает разворот («думал — мечта; оказалось — ловушка») → вопрос «как именно ловушка?» удерживает до конца

Шаблоны хуков под разные мифы:
- **Ставка-сюрприз:** «Представьте: <желанная сила>. <Герой> думал — это <позитив>. Оказалось — <ловушка>.» (Мидас, Прометей)
- **Ставка-ужас:** «Каждые <X> лет <страна> отправляла <жертв> на верную гибель. Назад не возвращался никто.» (Тесей, Минотавр)
- **Невозможный вопрос:** «Что если <сделать невозможное>? Один <герой> попытался — и <последствие>.» (Сизиф, Орфей)
- **Невероятная клятва:** «Один <бог> поклялся <обещание>. И сдержал слово, не подумав о последствиях.» (Аполлон+Кассандра, Зевс+Семела)

Этот хук — **отдельный абзац перед основным текстом**, в озвучке он становится `sentence_002` (после интро `sentence_001`). При разбивке `voiceover.md` на `voiceover/texts/sentence_NNN.txt` хук ОБЯЗАН быть отдельным файлом — это та фраза, под которую монтажёр кладёт самую сильную картинку первой сцены.

Шаблон с напоминанием про хук автоматически создаётся при «+ Новый миф» в webapp (см. README, раздел «Создание сценария через UI»).

## Персонажи (ЖЁСТКИЕ правила, применяются ко ВСЕМ сценариям)

1. **Все герои — антропоморфные бипедальные коты.** Не настоящие четвероногие кошки. Стоят, ходят, сидят, летают, жестикулируют как люди. Используют лапы как человеческие руки. Человеческие пропорции тела.
   - В каждом промпте обязательно: `anthropomorphic bipedal cat character, standing upright on two legs like a human, humanoid body proportions`

2. **В кадре НИКОГДА нет людей.** Ни на переднем плане, ни в толпе, ни как силуэты на фоне.
   - В каждом промпте обязательно негатив: `NO humans, NO people`

3. **В кадре НИКОГДА нет настоящих четвероногих кошек.** Модель без явного запрета часто рисует обычных кошек — особенно в сценах прыжков, полёта, бега.
   - В каждом промпте обязательно негатив: `NO real four-legged cats, only anthropomorphic bipedal cat characters`

4. **Консистентность персонажей внутри сценария.** Один и тот же герой во всех 25 сценах должен выглядеть одинаково — тот же окрас, возраст, борода, одежда, цвет глаз. Для каждого сценария в начале `scripts/*.md` заводится блок «карточка персонажей» в HTML-комментарии, и описание копируется в каждый промпт дословно. Нельзя писать «молодой кот» в одной сцене и «старый с бородой» в другой.

5. **Динамичные сцены — критическая зона.** Прыжки, полёт, бег, падения — именно там модель скатывается в «настоящих кошек на четвереньках». В таких промптах отдельно прописывать позу тела:
   - Прыжок: `human-like diving pose with arms outstretched, body upright not on four legs`
   - Полёт: `gliding pose like Superman with arms forward, body horizontal, humanoid posture`
   - Падение/паника: `humanoid body flailing arms, human-like gestures`

**Почему правила такие:** генерация icarus.md (первая версия) показала три типичных сбоя:
- (а) Дедал мутировал между сценами — в одной старик с бородой, в другой молодой кот
- (б) в сценах 8/17/19/20/21 герои были нарисованы как настоящие кошки на четырёх лапах
- (в) в сценах 13/14/16/19 модель добавила людей в кадр

Все три проблемы лечатся только явными формулировками в тексте каждого промпта — модель не помнит, что было в предыдущей сцене.

## Длина видео и структура сцен

- **Длина финального видео: 1:00–1:10** (YouTube Shorts / TikTok / Reels)
- На каждый сценарий — **20–25 коротких сцен по 2–4 секунды**. Одна сцена = одна анимация.
- Закадровый текст сцены — 7–10 слов (≈2–4 сек озвучки)
- Короткие сцены = высокая смена кадров = хорошее удержание зрителя. Длинные статичные сцены убивают ретеншн.

## Связка озвучки и видео (вариант A — принят 2026-04-20)

**Правило:** одно предложение озвучки = одна «сцена». Внутри сцены может быть несколько визуальных шотов для динамики. Маппинг — через имя файла, без ручных таблиц и ffmpeg-склеек.

**Именование файлов:**
- Озвучка: `sentence_NN.mp3` (один TTS-запрос на целое предложение — интонация сохраняется). В `approved_sentences/` файлы могут быть `sentence_NNN_vX.mp3` после ручного выбора лучшего варианта — при рендеринге переименовать в `sentence_NN.mp3`.
- Видео: `scene_NN_M.mp4`, где `NN` — номер предложения (совпадает с `sentence_NN`), `M` — индекс шота (`01`, `02`, `03`...).
  - Короткое предложение → один шот: `scene_05_01.mp4`.
  - Длинное предложение → несколько шотов для ритма: `scene_09_01.mp4`, `scene_09_02.mp4`, `scene_09_03.mp4`.

**Сцена в pyCapCut (`scene_structure_<миф>.py`):**
```python
Scene(
    sid="NN",
    audios=["sentence_NN.mp3"],
    videos=[VideoShot(file="scene_NN_01.mp4"), VideoShot(file="scene_NN_02.mp4")],
    text="Субтитры сцены",
    transition_after="Прыжок и отскок",
)
```
Длительность сцены = длительность аудио. Шоты делят это время поровну.

**Частые ошибки:**
- НЕ размазывать одно предложение на две отдельные сцены с отдельными аудио. Если нужна визуальная смена — делай шоты `_01/_02` внутри одной сцены.
- НЕ класть два предложения в один `scene_NN`. Если для двух коротких фраз хочется единый визуал («Заново. Навсегда.»), склей их в одно предложение в `voiceover.md` — ElevenLabs нормально тянет две точки в одном запросе. Интонация ломается только внутри *незакрытого* предложения.
- НЕ делать ffmpeg-concat на аудио. При этой схеме склейки не нужны.

## Визуальное разнообразие (важно!)

- **НЕ повторять одни и те же объекты** из сцены в сцену (вазы с кошками + статуи с кошками в каждом кадре — плохо).
- Варьировать окружение по ходу истории: греческие корабли (триремы), оливковые рощи, морские пейзажи, интерьеры мастерских, крепости, башни, храмы, рынки, пастухи, дельфины, чайки, облака, закаты, рассветы, пещеры, костры, свечи, факелы, масляные лампы.
- Разные ракурсы и планы: крупный план, средний, общий, вид сверху, от первого лица, низкий ракурс.
- Разное освещение: утро/полдень/закат/ночь/огонь/луна.
- **Кошачьи декоративные мотивы** (статуи котов, вазы с котами, фрески с котами) — уместны в **дворцах и храмах**, но НЕ в природных, производственных, морских сценах. В мастерской, на корабле, на скале — обычный греческий антураж без принудительных кошачьих деталей. Главные герои — всегда коты, но декор вокруг — по контексту.

## Уникальный subject-маркер в начале каждого промпта

**Правило:** первые 3–4 слова каждого `**Промпт:**` в `prompts/images.md` и `prompts/video.md` — это короткая английская фраза-действие, **уникальная между всеми сценами одного сценария**. Только потом идёт стилевой каркас.

**Пример:**
```
**Промпт:** persephone gathering spring flowers, highly detailed pixel art, 9:16 vertical, idyllic medium-wide shot...
**Промпт:** demeter torch night search, highly detailed pixel art, 9:16 vertical, sweeping cinematic wide shot...
**Промпт:** hades scooping persephone meadow, highly detailed pixel art, 9:16 vertical, dynamic medium shot...
```

**Зачем:** Google Flow / ImageFX берут первые слова промпта в имя экспортируемого файла (`Persephone_gathering_spring_flowers_<date>.jpeg`). Если префиксы уникальны — раскладка экспортированного архива по `scene_NN/vN.jpg` становится **тривиальной** (один pattern-matching без визуальной сверки). Если префиксы повторяются — архив превращается в кашу, в которой 12 файлов с одним именем покрывают 3 разные сцены и приходится открывать каждую картинку и сличать с промптом.

**Когда применять:** ВСЕГДА при написании `prompts/images.md` и `prompts/video.md` для нового сценария. Особенно критично если планируется **ручная** генерация в Flow (минуя `imagefx_runner.py` / `video_runner.py`).

**Как формулировать маркер:**
- 3–4 английских слова, описывающих что в кадре (`hades scooping persephone meadow`, `frozen river winter valley`, `pomegranate seed binding magic`)
- Не должен совпадать с маркерами других сцен — даже частично; избегать одинакового первого слова там, где можно (если первое слово неизбежно повторяется — `persephone …`, `hades …` — то 2-е и 3-е должны различаться)
- Маркер сам по себе должен быть **валидным описанием сцены**, потому что он попадает в начало промпта и слегка влияет на генерацию

**Проверка перед запуском раннера/Flow:**
```bash
grep '^\*\*Промпт:\*\* ' prompts/images.md \
  | sed -E 's/^\*\*Промпт:\*\* ([^,]+),.*/\1/' \
  | sort | uniq -c | sort -rn
```
Все строки должны быть с числом `1`. Если есть `2` или больше — переписать дубликат.

## Ограничения платформ (TikTok / YouTube Shorts)

Тематика греческих мифов часто задевает «триггерные» слова модерации. Правила ниже применять ко **всем** сценариям — к озвучке, субтитрам, промптам картинок и видео.

### Слова-триггеры в озвучке и субтитрах

TikTok распознаёт речь и ловит стоп-слова даже в аудио. YouTube мягче, но при прямых формулировках может снизить охваты Shorts или поставить возрастное ограничение (18+) — для коротких форматов это почти смерть ролика.

**Заменять:**

| Вместо | Использовать |
|---|---|
| убил / убивает | сразил, одолел, поверг, сокрушил |
| смерть | конец, гибель, уход, исчезновение |
| умер / погиб | пал, сгинул, не стало, покинул мир |
| труп | тело, поверженный |
| кровь | — (не упоминать в тексте) |
| самоубийство, повесился и т.п. | — (полностью переформулировать сцену) |

Контекст мифологии/истории/образования смягчает модерацию на YouTube, но TikTok-бот контекст не читает — поэтому пишем мягче по умолчанию.

### Визуальные ограничения (промпты картинок и видео)

Pixel art стиль сам по себе снижает риски (мультяшность), но прямые сцены насилия всё равно ловятся:

- **Кровь, красные брызги, раны** — не включать в промпты, добавлять в негатив: `no blood, no gore, no wounds`
- **Отрубленные головы, расчленение** — показывать как силуэт в мешке, тень, закадровый намёк. Голова Медузы — мешок/тень, не сам артефакт крупным планом
- **Момент удара / гибели** — не добивание в кадре. Только: вспышка, силуэт поверженного, падение, растворение в тени, затемнение
- **Исчезновение персонажа** (Эвридика, превращения) — мягкое растворение, частицы, туман, без агонии

### Приоритет сценариев по безопасности охватов

- **Самые безопасные:** Мидас (нет насилия), Арахна (превращение, без смерти), Нарцисс
- **Средний риск:** Орфей и Эвридика (эмоционально, но без резни), Персефона
- **Требуют аккуратной подачи:** Персей (Медуза), Тесей (Минотавр), Геракл (чудовища)

### 1. Ящик Пандоры

Боги владели огнём, но не делились им с людьми. И тогда Прометей решился на невозможное. Он тайно поднялся на Олимп и похитил огонь — символ силы, знаний и жизни.

Он передал его людям. И всё изменилось: люди научились готовить, греться и создавать орудия.

Зевс был в ярости! Чтобы проучить людей, он задумал коварный план и приказал Гефесту смешать обычную землю с водой и вылепить из этой смеси девушку невероятной красоты. Так появилась Пандора.

Зевс преподнёс Пандору как «идеальный дар» и направил её прямо к Прометею. Но Прометей недаром носил имя «Провидец»: он резко отверг её и строго наказал брату не принимать никаких даров с Олимпа.

Эпиметей же, чьё имя означает «Думающий после», ослушался. Очарованный Пандорой, он без колебаний решил, что она — лучшее, что когда-либо случалось в его жизни.

Пандора и Эпиметей женятся. Зевс дарит молодожёнам роскошный ларец. Но есть одно «но»: открывать его строго запрещено. «Это для вашего же блага», — говорит Зевс, прекрасно зная, что запретный плод — самый сладкий.

Любопытство сильнее запретов! Пандора решает: «Только один разок взгляну». БАМ! Из ящика вырываются болезни, войны и все беды мира. Зевс ликует — план мести сработал!

Мир изменился навсегда, а на самом дне осталась лишь Надежда.

## Озвучка (TTS через ElevenLabs)

**Сервис:** https://elevenlabs.io/app/speech-synthesis/text-to-speech

### Правила озвучки

1. **Озвучка идёт по предложениям, не по частям.** Если разбить одно предложение на два TTS-запроса, интонация в середине фразы ломается — ElevenLabs не знает, что предложение продолжается. Минимальная единица — полное предложение с точкой/восклицательным/вопросительным знаком.

   **Исключение — интро.** Первая строка сценария («Сизи́ф. Миф за минуту.», «Ика́р и Деда́л. Миф за минуту.» и т.п.) озвучивается ОДНИМ запросом — два коротких предложения, но читаются как единая титульная фраза.

2. **На каждое предложение — 3 варианта озвучки.** После Generate нажимаем Regenerate ещё 2 раза → получаем 3 mp3. На бесплатном тарифе ElevenLabs 3 варианта бесплатно, 4-й уже платный — не генерируем больше. Потом вручную выбираем лучший.

3. **Связь «предложение ↔ сцена» не 1:1.** В `voiceover.md` предложения могут:
   - совпадать со сценой (`сцена_03.mp3` = одно предложение),
   - содержать несколько сцен (одно длинное предложение покрывает `сцена_04` + `сцена_05`),
   - одна сцена может состоять из нескольких предложений (`сцена_18_01.mp3`, `сцена_18_02.mp3`).

4. **Структура папок:** каждое предложение хранится в своей подпапке со всеми 3 вариантами:
   ```
   content/<Название мифа>/voiceover/audio/
   ├── sentence_001/
   │   ├── sentence_001_v1.mp3
   │   ├── sentence_001_v2.mp3
   │   └── sentence_001_v3.mp3
   └── sentence_002/
       └── ...
   ```
   После выбора лучшего варианта вручную переименовать в `scene_NN.mp3` / `scene_NN_XX.mp3`.

5. **Ударения в тексте** (`Ика́р`, `Деда́л`, `Сизи́ф` и т.д.) вставляются прямо в поле ввода ElevenLabs — модель их правильно интерпретирует.

### Процесс

1. Скрипт `automation/elevenlabs_runner.py` читает `content/<Название мифа>/prompts/voiceover.md`, разбивает текст на предложения.
2. Для каждого предложения: открывает форму → вставляет текст → жмёт Generate → ждёт генерации → скачивает mp3.
3. Файлы складываются в `content/<Название мифа>/voiceover/audio/` с именами `sentence_001.mp3`, `sentence_002.mp3`, ...
4. **Ручной шаг:** пересопоставить предложения со сценами и переименовать в `scene_NN.mp3`.

## Инструкция по гуманизации текста

При написании и редактировании сценариев избегать:

- Канцелярита: «в рамках», «осуществлять», «данный», «является»
- ИИ-клише: «стоит отметить», «важно подчеркнуть», «нельзя не упомянуть»
- Раздувания значимости: «ключевой», «непреходящий», «знаковый»
- Списков с жирными заголовками там, где уместен живой текст
- Шаблонных концовок: «будущее выглядит многообещающим» и т.п.
- Подхалимского тона и артефактов чатбота

Добавлять:
- Живой ритм: чередовать короткие и длинные предложения
- Конкретику вместо абстракций
- Иронию и характер там, где уместно
- Русские кавычки «ёлочки», не английские "..."
